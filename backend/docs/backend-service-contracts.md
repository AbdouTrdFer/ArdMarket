# Backend Service Contracts

This document defines the currently implemented HTTP contracts and header contracts between services.

## 1) Shared Header Contract

### 1.1 Headers produced by auth validation

When `auth-service` validates a token successfully, it returns:

- `X-User-Id`: authenticated user id
- `X-User-Roles`: comma-separated roles (example: `ADMIN,USER`)
- `X-User-Email`: authenticated user email

These headers are injected by Traefik (`forward-auth`) into protected upstream requests.

### 1.2 Header used for trusted internal calls

- `X-Request-Source: auth-service`

`user-service` currently bypasses role/header checks for requests carrying this exact source value.

## 2) JWT Contract (Auth-Service Issued Token)

The internal JWT generated by `auth-service` includes:

- `sub`: user email
- `iss`: `auth-service` (or configured issuer)
- `id`: user id
- `email`: user email
- `roles`: list of role names
- `iat`, `exp`

Validation checks include:

- Subject must exist
- `email` claim must exist
- `sub` must match `email`
- `id` claim must exist
- `roles` must be non-empty
- issuer must match configured issuer (if present)

## 3) Endpoint Contracts

## 3.1 Auth Service

Base path: `/auth`

### `POST /auth/login`

Purpose:

- Accept Google `id_token`
- Validate with Google tokeninfo
- Verify allowed email domain
- Fetch existing user from `user-service`
- Return internal JWT access token

Request body:

```json
{
  "idToken": "google-id-token"
}
```

Success response (`200 OK`):

```json
{
  "accessToken": "<internal-jwt>"
}
```

Typical failure responses:

- `400`: missing `idToken`
- `401`: authentication failure
- `403`: unauthorized domain or disabled account

Example:

```bash
curl -X POST http://localhost/api/auth/login \
  -H "Content-Type: application/json" \
  -d '{"idToken":"<google-id-token>"}'
```

### `GET /auth/validate`

Purpose:

- Validate internal JWT
- Return identity/role headers for downstream services

Request headers:

- `Authorization: Bearer <internal-jwt>`

Success response (`200 OK`):

- Empty body
- Response headers include:
  - `X-User-Id`
  - `X-User-Roles`
  - `X-User-Email`

Failure response (`401 Unauthorized`):

- Plain text message (current behavior)

Example:

```bash
curl -i http://localhost:8081/auth/validate \
  -H "Authorization: Bearer <internal-jwt>"
```

## 3.2 User Service

Base path: `/users`

### `POST /users/auth-user`

Purpose:

- Internal endpoint used by `auth-service` login flow
- Returns existing user record
- On first login, updates `firstLogin=false` and `pfpUrl`

Request headers:

- `X-Request-Source: auth-service`

Request body:

```json
{
  "email": "user@ump.ac.ma",
  "pfpUrl": "https://example.com/avatar.jpg"
}
```

Success response (`200 OK`):

- Returns user entity JSON (includes id, email, roles, enabled, firstLogin, and other entity fields)

Failure responses:

- `400`: email missing
- `404`: user not found

Direct internal example:

```bash
curl -X POST http://localhost:8082/users/auth-user \
  -H "Content-Type: application/json" \
  -H "X-Request-Source: auth-service" \
  -d '{"email":"user@ump.ac.ma","pfpUrl":"https://example.com/avatar.jpg"}'
```

### `GET /users/{id}`

Purpose:

- Retrieve user information
- Requires `ADMIN` role (`@RequireRole({"ADMIN"})`)

Required forwarded headers:

- `X-User-Id`
- `X-User-Roles`
- `X-User-Email` (optional for check, useful for context)

Success response (`200 OK`):

```json
{
  "email": "ayoubelebbar@ump.ac.ma",
  "firstName": "Ayoub",
  "lastName": "Elebbar",
  "pfpUrl": "...",
  "roles": ["ADMIN"],
  "enabled": true,
  "firstLogin": false
}
```

Failure responses:

- `401`: missing auth headers
- `403`: insufficient roles
- `404`: user not found

Gateway example (recommended):

```bash
curl http://localhost/api/users/1 \
  -H "Authorization: Bearer <internal-jwt>"
```

## 4) Inter-Service Call Contract (Auth -> User)

Current implemented call:

- Caller: `auth-service`
- Client type: OpenFeign
- Target: `${services.user-service.url}/users/auth-user`
- Header injected automatically: `X-Request-Source: auth-service`
- Body fields sent: `email`, `pfpUrl`

Expected behavior:

- `user-service` must return an existing user for valid domain accounts.
- `auth-service` expects fields at least compatible with:
  - `id`
  - `email`
  - `roles`
  - `enabled`
  - `firstLogin`

## 5) Notes on Response Shapes

Current behavior is partially consistent but not fully standardized:

- Some auth errors return structured JSON (`status`, `message`)
- Some auth validation failures return plain text
- user-service returns plain text for some errors via servlet `sendError`

For new endpoints, follow the target error standard in `backend/docs/backend-service-rules.md`.
